# Plan: Кликабельные ссылки в карточках

## Цель

Сделать кликабельными **все** ссылки в тексте задачи — клик открывает URL в
браузере по умолчанию. Работает одинаково во всех терминалах, не завися от их
собственного автоопределения URL.

Что считается ссылкой и где кликабельно:
- **Markdown-ссылки** `[текст](url)` — кликабелен и inline-текст в теле задачи,
  и строка footnote `[n] текст: url`.
- **Голые URL** (`https://…`, `http://…`), набранные прямо в тексте — кликабелен
  сам URL inline (footnote для них не создаём — URL и так виден).

---

## Контекст и почему так

Сейчас «ссылки» — только markdown-формат `[текст](url)`, который `parse_links`
(`src/links.rs`) превращает в footnotes (`render_with_footnotes`). Голые URL в
тексте не распознаются вообще.

Рассмотренные альтернативы для кликабельности и почему отклонены:

- **Автоопределение URL терминалом.** Не работает: включён захват мыши
  (`EnableMouseCapture`, `src/main.rs`). В mouse-reporting терминал отдаёт клики
  приложению, его URL-детект отключается. Мышь нужна (выбор карточек, скролл) —
  отключить нельзя.
- **OSC 8 гиперссылки.** ratatui 0.29 не поддерживает. Через `Span` escape не
  выдать — ширина считается по графемам, escape-байты ломают раскладку.

Вывод: клики и так приходят в приложение — обрабатываем их сами. Запоминаем
экранные координаты ссылок (по колонкам) и по клику открываем URL.

### Ключевое решение: ручной перенос вместо `Paragraph::wrap`

Сейчас текст карточки рисуется `Paragraph` с `Wrap { trim: false }` — это
**пословный** перенос ratatui. Он не совпадает с посимвольным расчётом высот в
`wrapped_rows` (потенциальный рассинхрон высот уже сейчас). Для точного hit-test
по колонкам геометрия должна быть детерминированной, поэтому переходим на
**ручной посимвольный перенос** текста карточки — так же, как уже сделано в
редакторе (`split_visual_chunks` в `src/ui.rs`). Тогда позиция каждого символа на
экране вычисляется тем же кодом, что и рендер, и hit-test точен.

## Шаг 1 — Детектирование ссылок (`src/links.rs`)

Расширить разбор так, чтобы он находил и markdown-ссылки, и голые URL.

- Добавить детект голых URL: сканировать текст на `http://` / `https://`,
  набирать URL до первого пробела/перевода строки/закрывающего разделителя
  (отсекать хвостовые `).,;` и т.п.). Не захватывать URL, уже входящие в
  `[..](..)`-разметку (их обрабатывает markdown-ветка).
- Расширить `RenderedText`, чтобы помимо `display`/`footnotes` отдавать список
  **inline-кликзон** в координатах строк `display` (логических, до переноса):

```rust
pub struct InlineLink {
    pub line: usize,        // индекс строки в display (по '\n')
    pub col_start: usize,   // в символах от начала строки
    pub col_end: usize,     // эксклюзивно
    pub url: String,
}
```

  Для markdown-ссылки `col_start..col_end` покрывает inline-текст (`текст`, без
  маркера `[n]`). Для голого URL — сам URL.
- `footnotes` остаются как есть (для markdown-ссылок) — клик по строке footnote
  тоже открывает URL (см. Шаг 3).

## Шаг 2 — Ручной рендер текста карточки (`src/ui.rs`)

В `draw_task_list` заменить `Paragraph { Wrap }` на ручной посимвольный перенос:

- Текстовые строки `display` резать на визуальные чанки шириной `inner_w`
  (переиспользовать/повторить логику `split_visual_chunks`).
- Сохранить текущую раскраску: маркеры `[n]` и footnotes красятся по
  `LINK_PALETTE` (вынести логику из `build_display_line`, чтобы работала
  поверх чанкования).
- Высоты карточек (`card_heights`) считать тем же чанкованием — рассинхрон с
  рендером уходит.
- Убрать `.wrap(Wrap { trim: false })`; строки уже разбиты вручную.

(Это самый объёмный шаг — фактически переписываем построение строк карточки.)

## Шаг 3 — Геометрия кликзон (`src/app.rs` + `src/ui.rs`)

Поле в `App` (перестраивается каждый кадр, как `card_rows`):

```rust
/// (row, col_start, col_end, url) для каждой видимой кликзоны, координаты терминала;
/// одна ссылка может дать несколько прямоугольников, если переносится.
pub link_rects: Vec<(u16, u16, u16, String)>,
```

При ручном рендере (Шаг 2) для каждой `InlineLink` и каждой строки footnote
вычислять экранные прямоугольники:
- inline-ссылка: её символы `col_start..col_end` могут попасть в разные чанки →
  на каждый затронутый визуальный ряд свой прямоугольник
  `(row, x_start, x_end, url)`. Базовая колонка контента — `card_rect.x + 1`.
- footnote: вся строка целиком (как в первой версии плана), все её визуальные
  ряды.
- Клиппинг по `[card_rect.y, card_rect.y + render_h)` и по ширине; ряды вне
  видимой части не регистрируем.

## Шаг 4 — Открытие URL (`src/input.rs`)

```rust
fn open_url(url: &str) {
    // только http/https — UX и защита от инъекции в shell (особенно Windows start)
    if !(url.starts_with("http://") || url.starts_with("https://")) {
        return;
    }
    #[cfg(target_os = "macos")]
    let _ = std::process::Command::new("open").arg(url).spawn();
    #[cfg(target_os = "windows")]
    let _ = std::process::Command::new("cmd").args(["/C", "start", "", url]).spawn();
    #[cfg(target_os = "linux")]
    let _ = std::process::Command::new("xdg-open").arg(url).spawn();
}
```

URL передаётся отдельным аргументом (не через shell-строку); схема валидируется.
Ошибки игнорируем.

## Шаг 5 — Обработка клика (`src/input.rs`)

В `handle_normal` и `handle_trash`, в ветке `Down(Left)`:
1. Сначала искать попадание `(row, col)` клика в `app.link_rects`
   (`row` совпадает и `col_start <= col < col_end`) → `open_url(url)` и `return`.
2. Иначе — прежний выбор карточки по `card_rows`.

В Trash footnotes/URL рендерятся серым, но ссылки оставляем кликабельными.

## Шаг 6 — Проверка

- [ ] Клик по голому URL в тексте открывает браузер (macOS — локально).
- [ ] Клик по inline-тексту markdown-ссылки открывает её URL.
- [ ] Клик по строке footnote `[n]` открывает тот же URL.
- [ ] Клик мимо ссылок выбирает карточку, как раньше.
- [ ] Перенос: ссылка, разорванная на 2 ряда, кликабельна на обоих рядах.
- [ ] Скролл и частично обрезанные карточки — координаты кликзон корректны.
- [ ] Раскраска маркеров/footnotes и высоты карточек не сломались после перехода
      на ручной перенос.
- [ ] Windows: открытие через `start`, URL с кавычками не ломает команду.
